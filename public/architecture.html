03 — วิธีคิดของ ML

ทำไมโมเดลถึงเชื่อถือได้

ไม่ใช่ deep learning กล่องดำ — เป็น classical ML บนข้อมูลตาราง ออกแบบรอบความซื่อตรงทางสถิติ. หัวใจ 4 ความคิด:

① พยากรณ์ "ความน่าจะเป็น" ไม่ใช่ใช่/ไม่ใช่

target รายสัปดาห์ = มีวัน heatwave ≥1 วัน ในหน้าต่าง 7 วัน ที่ lead 2–6 สัปดาห์. โมเดลคืน p ∈ [0,1]

y_week_L = 1 if any heatwave-day in week@(issue + L·7d)

โมเดล pooled ตัวเดียวเรียนข้ามทุกจังหวัด (~828k แถว) — จังหวัดที่เหตุการณ์น้อยได้สกิลจากการแชร์

② Features = antecedent-only (กัน leakage)

ทุก feature คำนวณจากข้อมูล ≤ วันออกพยากรณ์เท่านั้น

// local (ราย cell) sm1/sm3 mean7/mean30/trend, tmax_mean7 in_hw_today (trailing streak ≥3), hot_frac7 // shared mjo_phase/amp, nino34_lag1m, doy_sin/cos // static lat, lon, region (one-hot×6)

ห้ามใช้ค่าวันเป้าหมาย/base_rate ดิบ → กัน target leak

③ Calibration — สำคัญที่สุด

LightGBM ดิบมี AUC ดี (0.69–0.76) แต่ BSS ติดลบทุก lead — เพราะลำดับดีแต่ตัวเลข p ไม่ตรงความเป็นจริง. แก้ด้วย Platt scaling บน block แยกเวลา

p ดิบ (overconfident) p หลัง calibrate เส้นประ = ความถี่จริง
p=0.2
p=0.4
p=0.6

เมื่อโมเดลบอก "0.6" มันควรเกิดจริงราว 60% ของครั้ง — calibration ดัน BSS เป็นบวก ชนะ baseline ได้

④ Validation: ต้องชนะ baseline อย่างมีนัยสำคัญ

ถ้าทายไม่ดีกว่า "อากาศปกติตามฤดู" ก็ไร้ประโยชน์ เกณฑ์ผ่าน = ชนะ climatology และ persistence (paired test, q<0.05)

โมเดลlead 2lead 3lead 4lead 5lead 6
logistic_bal_cal +0.168+0.109+0.118+0.050−0.014
lgbm_cal +0.144+0.156+0.059+0.024+0.103
persistence (baseline) +0.095+0.048 +0.047+0.024+0.017

BSS เทียบ seasonal climatology (regional p90). ยิ่งบวกยิ่งดี. Feature สำคัญสุด: nino34_lag1m > sm1_mean30 — ตรงสมมติฐาน ENSO + ความจำความชื้นดิน

⑤ จาก probability → ป้ายความเสี่ยง

risk_level() ใช้ อัตราส่วนต่อ base_rate ของจังหวัด — "เสี่ยงกว่าปกติกี่เท่า" ไม่ใช่ตัด p ตรงๆ

Low
p<0.05 หรือ ratio<0.75
Normal
ratio < 1.5
Elevated
ratio < 2.5
High
ratio ≥ 2.5
ตัวอย่างจาก contract: กรุงเทพฯ lead 2 → p=0.386, base_rate=0.114, ratio=3.4 → High
Risk keys เหล่านี้เป็น immutable — Frontend อ่านตรงๆ ห้ามเปลี่ยน Low/Normal/Elevated/High

// deepseekContract.ts — อ่านแล้ว validate ก่อนใช้เสมอ
export function assertContract(c: unknown): ForecastContract {
  if (c.schema_version !== 1)
    throw new Error(`schema_version ไม่รองรับ: ${c.schema_version}`);
  if (c.n_provinces !== c.provinces.length)
    throw new Error(`n_provinces ไม่ตรง (${c.n_provinces} vs ${c.provinces.length})`);
  return c as ForecastContract;
}

export function mapPoints(c): MapForecastPoint[] {
  return c.provinces.map(p => {
    const f = p.forecasts.find(x => x.lead_weeks === 2);
    return { province_id: p.id, lat: p.lat, lon: p.lon,
      risk_level: toRisk(f.risk_level_en),   // "High" → 'High'
      probability: f.probability,
      target_date: addDaysISO(p.issue_date, f.lead_weeks * 7) };
  });
}

สถาปัตยกรรม i18n + Font Scaling

i18n ผ่าน t() hook

ทุกข้อความในแอปใช้ t(key) จาก useSettings — ไม่มี hardcode ภาษาไทย/อังกฤษใน JSX

// ✅ ถูก <ScaledText>{t('riskAlerts')}</ScaledText> // ❌ ผิด (hardcode) <ScaledText>แจ้งเตือนความเสี่ยง</ScaledText>

Font Scaling ผ่าน fontSize prop

ScaledText ต้องส่งขนาดผ่าน fontSize prop — ถ้าใส่ใน style.fontSize จะ override ค่า scale

// ✅ ถูก — scale ตาม setting <ScaledText fontSize={14} style={[styles.label]}>...</ScaledText> // ❌ ผิด — style override ค่า scale <ScaledText style={[{ fontSize: 14 }]}>

FONT_SCALE: small=0.85 · default=1.0 · large=1.25

06 — Automation & deployment

ระบบอัตโนมัติ ไม่ต้องรัน manual

ทุกอาทิตย์ Backend รันเอง Frontend deploy เอง — ผู้ดูแลไม่ต้องทำอะไรในสภาวะปกติ

1
GitHub Actions · ทุกวันจันทร์ 09:00 น. ไทย (02:00 UTC)
ดาวน์โหลด ERA5 ล่าสุด + รีเฟรช MJO/Niño3.4
Secrets: CDSAPIRC (CDS API key) ตั้งใน repo settings — ไม่มีใน code
2
GitHub Actions · ต่อเนื่อง
ทำนาย + validate + publish ไป heatwave-contract
Secrets: GH_CONTRACT_TOKEN (PAT write access ใน heatwave-contract)
Hard gate: ถ้า validate fail → abort → ไม่ commit → เว็บแสดงข้อมูลรอบก่อน
3
GitHub Actions · commit docs/forecast_provinces.json กลับ repo หลัก
git pull --rebase + push (retry loop 3 ครั้ง)
Retry loop ป้องกัน "fetch first" reject กรณี user push ระหว่าง workflow รัน
4
Vercel · trigger อัตโนมัติเมื่อ push ถึง HeatMAP_Frontend/main
Build Expo web export → deploy
command: bunx expo export -p web --clear
ต้อง --clear เพื่อ flush Metro cache + ฝัง EXPO_PUBLIC_FORECAST_URL ใหม่
5
ผู้ใช้ทั่วไป
เปิดเว็บ → fetch forecast_provinces.json จาก GitHub Pages
ไม่มี backend server — ทุกอย่างเป็น static. ถ้า CDS ล่มรอบนี้ ผู้ใช้ยังเห็นพยากรณ์รอบก่อน
